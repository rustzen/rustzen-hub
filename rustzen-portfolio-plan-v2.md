# Rustzen 仓库、产品与授权整合执行方案（RFC v2）

状态：候选方案，未经执行授权  
日期：2026-07-13  
范围：仓库命名、产品组合、私有 Monorepo 迁移、公开桌面模板、授权边界  

## 1. 决策摘要

采用以下唯一命名与职责：

| 名称 | 类型 | 可见性 | 唯一职责 |
|---|---|---|---|
| `rustzen-hub` | 服务与门户仓库 | 站点公开，后台边界独立 | 官网、下载入口、授权与发布服务权威 |
| `rustzen-admin` | 工程模板仓库 | 公开 | Web/Rust 工程模板，不属于产品套件 |
| `rustzen-core` | 中立基础库仓库 | 公开 | 至少两个真实消费者验证过的产品无关能力 |
| `rustzen-tools` | 产品源码 Monorepo | 私有 | Clear、Clipboard、Zipper 的内部源码边界 |
| `rustzen-desktop-template` | 桌面模板仓库 | 公开 | 新建、可复用的 Rust/Tauri 客户端模板 |
| `rustzen-launcher` | 原型仓库 | 当前状态不变 | Launcher/搜索原型，另行决定未来 |
| Rustzen Suite | 对外产品家族 | 公开品牌 | Clear、Clipboard、Zipper 的集合名称 |
| Rustzen Pro | 商业授权计划 | 公开名称 | 解锁符合条件的 Suite 能力，不是应用或统一版本 |

`rustzen-tools` 不作为对外品牌；公开页面不得使用“Rustzen Tools”指代产品。`rustzen-desktop-template` 不属于 Rustzen Suite，不是 Launcher 替代品，也不是 Suite 的开源实现或共享运行时。

## 2. 目标与成功标准

目标不是减少仓库数量本身，而是降低一个人维护三个产品时重复修改 CI、发布流程、依赖与授权客户端的成本，同时保持产品独立发布和故障隔离。

最终成功必须同时满足：

1. Clear、Clipboard、Zipper 可从干净环境分别安装依赖、构建、测试、打包和发布；构建一个产品不要求构建另外两个。
2. 三者继续拥有独立版本、标签、二进制/包名、Bundle ID、签名凭据、更新通道、数据目录和回滚/恢复方案。
3. 一个产品的变更不会触发其他产品的发布；共享变更必须验证所有声明的消费者。
4. Rustzen Pro 后端故障或客户端离线时，本地免费能力和用户数据仍可使用；三产品不要求同步发版。
5. 迁移前后可量化比较维护成本；没有达到预设收益时停止迁移。
6. 公开模板只依赖公开来源，删除示例和实验后仍能独立构建和使用。

## 3. 明确不做的事项

- 不把三个产品合成一个可执行文件、统一安装器或强制共享更新器。
- 不建立统一产品版本或同步发布列车。
- 不因迁移而更改包名、Bundle ID、数据目录、签名/更新密钥或公开下载地址。
- 不把 Zipper 改造成 Tauri 应用。
- 不在第一阶段建立根 Cargo/npm Workspace 或统一锁文件。
- 不把路径过滤当作唯一 CI 正确性边界。
- 不让 `rustzen-tools` 成为通用杂物仓库或授权/发布服务权威。
- 不把商业、授权策略或产品标识放入 `rustzen-core`。
- 不从 `rustzen-launcher`、Clear、Clipboard、Zipper 复制实现到公开模板。
- 不在一次切换中迁移三个产品，不保留两个可写源码权威或两个生产发布者。
- 不自动归档旧仓库；归档需要单独授权。

## 4. `rustzen-tools` 初始结构

第一阶段采用“联邦式 Monorepo”：同一 Git 仓库，多个自治产品工作区。

```text
rustzen-tools/
├── products/
│   ├── clear/          # 保留自身 Cargo/npm/Tauri 结构和锁文件
│   ├── clipboard/      # 保留自身 Cargo/npm/Tauri 结构和锁文件
│   └── zipper/         # 保留 Rust CLI/npm 结构和锁文件
├── shared/             # 通过准入评审后才可进入
├── tooling/            # 仅编排产品本地命令
├── manifests/
│   ├── products.*      # 产品身份、命令、发布单元
│   ├── ownership.*     # 路径所有权
│   └── ci-impact.*     # 依赖与影响图
├── docs/
└── README.md
```

根工具只能计算影响范围、调用产品本地命令、校验清单和生成汇总，不拥有产品版本、锁文件或发布身份。以后是否统一 Workspace 必须作为独立提案验证，不能视为 Monorepo 的默认结果。

共享包准入条件：至少两个真实消费者、产品中立语义、明确负责人、稳定接口、独立测试、消费者清单、变更策略、无循环依赖，并能证明减少了重复维护。产品特有逻辑保留在 `products/<name>`。

## 5. 依赖与权威方向

```text
rustzen-desktop-template ──可选──> 已发布的公开 rustzen-core API
rustzen-tools ─────────────允许──> 已发布或精确版本的公开 rustzen-core API
rustzen-tools 产品客户端 ────────> rustzen-hub 的版本化网络契约
rustzen-core ────────────────────> 不依赖 Hub 产品策略、Tools 或模板
rustzen-desktop-template ────────> 不依赖私有仓库、商业授权或内部端点
```

授权协议与发布服务的权威始终在 `rustzen-hub`。`rustzen-tools` 中的 schema 只能是生成或固定版本的客户端输入。

## 6. Rustzen Pro 契约

授权使用稳定的 `product_id`、`feature_id` 和显式 `contract_version`；客户端与后端都声明支持的协议范围。缓存授权必须签名或经过等价认证。

`rustzen-hub` 必须拥有版本化的 Entitlement Runtime Policy。每条策略必须填写：`policy_version`、适用的 `contract_version` 范围、`product_id`、`feature_id` 或功能类别、离线宽限时长、时钟偏差容忍度、可信时间源、缓存验证规则、宽限期后行为、确认撤销后的行为、用户自有数据的读取/导出规则、激活与刷新恢复、过低/过高协议版本行为。具体时长不得在本 RFC 中臆定，但任何产品进入 G3 或 G5 前必须引用一份已批准且无空值的策略；“按策略”不能作为通过证据。

| 状态 | 付费能力 | 本地免费能力与数据 |
|---|---|---|
| 在线有效授权 | 按 feature ID 开启 | 完全可用 |
| 宽限期内有效缓存授权 | 开启 | 完全可用 |
| 后端不可用且从未激活 | 暂不能新激活 | 应用与免费能力可用 |
| 缓存超过宽限期 | 按引用的 Runtime Policy 执行 | 不删除、不损坏用户数据，并保留策略规定的读取/导出能力 |
| 未知 feature ID | 默认锁定 | 不受影响 |
| 不支持的协议版本 | 发布前阻断；运行时安全失败 | 不做破坏性操作 |
| 已确认撤销 | 按引用的 Runtime Policy 执行 | 用户自有数据仍可按策略读取/导出 |
| 时钟异常 | 按策略中的容忍度与可信时间源处理；无法确认则安全失败 | 不删除或损坏状态 |

对每个产品声明支持的协议范围，至少测试：最低客户端×最低后端、最低客户端×当前后端、当前客户端×最低后端、当前客户端×当前后端、低于范围、高于范围、每个独立产品版本引用的策略版本，以及离线启动、超时、过期/损坏缓存、撤销、未知功能和仅产品版本升级。范围不连续或存在能力协商时覆盖每个边界。任何产品可独立升级，不要求三产品同时发布。

## 7. 每产品独立迁移状态机

Clear、Clipboard、Zipper 分别推进：

```text
LEGACY_AUTHORITATIVE
→ IMPORTED_SHADOW
→ BUILD_PARITY_PROVEN
→ RELEASE_REHEARSAL_PROVEN
→ MONOREPO_SOURCE_AUTHORITATIVE
→ LEGACY_RELEASE_MIRROR
→ MONOREPO_RELEASE_AUTHORITATIVE
→ LEGACY_PUBLISHER_DISABLED
→ LEGACY_READ_ONLY
→ ARCHIVE_ELIGIBLE
```

源码权威与发布权威是两个决策。每次转换记录：前后权威、开启/关闭的工作流、凭据访问、精确源码 SHA、批准人、反向导出或恢复步骤。一个产品失败不得回退已验证的其他产品。

推荐试点顺序：Clipboard → Zipper → Clear。Clipboard 当前工作区干净且可验证 Tauri；Zipper 第二个验证异构 CLI/npm；Clear 只有在未提交改动和失效 upstream 得到无损处置后进入迁移。

## 8. 可逆执行阶段与门禁

### G0：命名、身份与基线清单

为每个产品记录：分支、HEAD SHA、工作区状态、upstream 状态、构建/测试/打包命令、标签、问题与发布 URL、包名、Bundle ID、二进制名、签名身份、更新密钥/Feed、制品名、数据目录、凭据所有者。所有未知项必须关闭或明确阻断，不能以假设通过。

Clear 的未提交内容只能：提交到获批分支、导出带 SHA-256 的补丁、经单独授权丢弃，或原地保留并明确排除。禁止隐式 stash、reset、clean、commit 或迁移。

通过：干净克隆或获批源码包能复现选定源码。  
停止：来源不可复现、更新地址或包所有权未知、无法构建。  
回退：不改仓库，丢弃清单尝试。

### G1：一次性导入 PoC

该 Gate 需要单独明确授权。在可删除的私有测试仓库导入历史，识别标签冲突。旧仓库不修改。每条映射必须记录：源仓库、ref、commit SHA、tree SHA、目标仓库、目标 commit SHA、目标路径、目标 subtree SHA、导入工具与版本、参数和记录 SHA-256；必须支持两个方向查询并独立验证 tree/subtree 等价性。

通过：文件完整、边界确定、双向可追溯。  
停止：历史不可追溯、边界模糊、需修改源仓库。  
回退：删除 PoC。

### G2：独立构建与 CI 影响验证

每产品从干净主机独立执行其现有命令。CI 使用声明式影响图：产品本地变更只跑自身；共享代码跑所有消费者；工具链、根工具、发布/签名工具按消费者展开；未知路径 fail closed 并跑完整矩阵。

设一个始终存在的必需检查 `monorepo-impact-validation`，确认所有必要 Job 实际运行。Fixture 至少覆盖产品本地、共享授权客户端、发布工具、根工具链、锁文件、未知路径、删除/重命名和多提交 merge-base。

通过：无依赖产品被漏测，产品本地变更不运行无关发布流程。  
停止：只靠路径过滤、required check 消失、Zipper 需要 Tauri、无关锁文件变化。  
回退：关闭 PoC CI，保留旧仓库流程。

### G3：身份、更新与非生产发布演练

默认只使用隔离测试包、测试 Feed、测试环境和测试凭据。验证隔离的旧安装 → 新构建更新、原有数据 → 新版本、npm 包名与 URL 不变、等价可信发布流程和旧 Release/资产可下载。真实生产客户端通过生产 Feed 的验证只属于 G5 有限发布。标签采用产品命名空间（例如 `clear/vX.Y.Z`），制品绑定产品与不可变 SHA。每产品凭据隔离，禁止仓库级共享签名/更新密钥。

读取或修改生产签名身份、更新密钥、npm 可信发布、OIDC subject、生产 Feed、凭据授权或发布环境都需要单独批准。对每个外部系统记录修改前状态、修改内容、修改后状态、所有者、恢复步骤、恢复验证及配置版本；平台不能同时容纳旧配置和演练配置时不得替换生产配置。

通过：草稿、staging 或获批 prerelease 可完成而不发布无关产品。  
停止：必须使用生产发布才能测试、来源证明失效、身份因迁库变化。  
回退：删除演练流程，并恢复所有外部注册表、OIDC、凭据授权、测试 Feed、环境权限和可信发布配置；恢复验证通过前 G3 不关闭。

### G4：单产品源码切换

该 Gate 需要单独明确授权。一次只切换一个产品。`rustzen-tools` 成为唯一人工可写源码权威；向一次性旧仓库克隆的反向导出测试通过。发布权仍在旧仓库时，必须在切换前批准并验证以下一种模式：A. G4 到 G5 期间禁止生产发布；B. 旧仓库成为 `LEGACY_RELEASE_MIRROR`，只接受由 `rustzen-tools` 权威提交确定性生成、单向且可审计的镜像写入，以运行旧发布流程。镜像禁止人工源码修改，也不得反向成为源码权威。两种模式未选定时 G4 不得通过。完整只读状态延后到 G5 禁用旧发布者之后。

通过：所有新增变更可反向导出，其他产品不必同时迁移。  
停止：双写、开放工作丢失、无法反向导出。  
回退：把已批准的新变更反向导回旧仓库，再恢复其源码权威。

### G5：单产品发布切换

需要单独批准。先禁用旧发布者，再启用新发布者；绝不同时可发布。有限/分阶段发布后验证旧客户端更新、包/Feed 兼容和数据恢复。

通过：只有新流程可发布且旧流程可恢复。  
停止：双发布者、旧客户端不能更新、来源证明失效、回退依赖未测降级。  
回退：停止继续分发，恢复旧发布权；客户端使用已测试降级、数据恢复或更高版本 forward-fix，未验证时不得宣称可降级。

### G6：分阶段观察与下一个产品

首个 PoC 前建立 measurement registry。每项指标记录 `metric_id`、范围、公式、单位、数据源、基线/比较窗口、最小样本、runner 硬件或镜像、工具链、缓存策略、阈值、零基线处理、负责人和证据路径。构建时间按产品分别统计干净/缓存构建，在同 runner 与固定工具链上至少 5 次，使用 median 与 p95；人工发布步骤只计人工动作/批准；CI fan-out 计每类变更启动的产品流水线数；配置文件减少与人工维护减少分开，基线为零时不使用百分比。至少一项主要人工维护指标必须改善，不能只凭文件数通过。

G6-A（Clipboard 试点）：至少 14 天和 1 次真实独立发布；无更新、数据、身份或发布权回归；无关产品 CI 为 0；反向导出与发布权恢复通过；Clipboard 人工发布步骤不增加；干净构建 median/p95 均不超过批准预算（默认上限为基线 +20%，如基线有效）；Monorepo 维护开销不超过预先批准上限。通过后 Zipper 才可进入 G4，不要求此时证明三产品 30% 收益。

G6-B（Clipboard+Zipper）：两者独立发布；共享变更消费者漏跑为 0；本地变更不触发另一产品发布；至少一个真实重复维护场景被共同工具消除；共享改动不引入同步发布；两产品人工维护收益达到 measurement registry 中预先批准的阶段阈值。通过后 Clear 才可进入 G4。

G6-C（最终组合）：所有计划产品完成观察窗后，以最终 30 天或更长窗口验证组合维护收益、CI、构建和发布预算。最终目标是主要人工维护指标改善至少 30%；未达到则停止进一步 Workspace 统一或共享，不自动回退已经安全且独立可维护的迁移。

### G7：旧仓库归档资格

只有源码/发布回退窗口结束、旧问题/Release/资产和链接仍可访问、无外部消费者依赖仓库特有行为、并获得单独归档授权后才具备资格。默认永久只读也可以。

## 9. `rustzen-desktop-template` 独立工作流

该仓库的创建与公开发布分别需要单独明确授权。仓库采用新历史和“公开来源创建”，不声称法律意义上的 clean room。输入仅限批准的公开来源；维护来源和依赖台账。不得把 Launcher 或私有产品源码作为输入。

建议结构：

```text
rustzen-desktop-template/
├── src/                 # 权威的最小前端模板
├── src-tauri/           # 权威的最小 Rust/Tauri 模板
├── examples/            # 可删除、非权威、仅 mock
├── experiments/         # 可删除、非生产实验
├── docs/
└── README.md
```

发布门禁：

1. 仅公开依赖，无私有 Git URL、相邻私有 Checkout、内部 Hub 端点、授权 Feature ID、签名/更新配置或生产凭据。
2. 使用新的中性示例 Bundle ID 和数据目录；执行 secret、依赖许可和内容相似性扫描。
3. 删除 `examples/` 与 `experiments/` 后，基线构建、测试、文档和模板使用仍通过。
4. README 明确：可复用模板；示例非权威；不是 Suite 产品、Launcher、生产发行基线或受支持的 Rustzen Desktop 二进制。
5. 公开模板默认不依赖 `rustzen-core`；只有基线确有必要且使用已发布公共 API 时才能引入。
6. 在公开前单独批准仓库许可证、版权声明、第三方 NOTICE、贡献政策、公开可见性和发布动作；依赖许可证扫描不能代替仓库自身许可决策。

发现私有依赖、产品标识、不可解释的复制实现或将示例作为产品发布时立即停止。该工作流与 `rustzen-tools` 迁移互不阻塞。

## 10. 历史与回滚的准确含义

- Git：保留来源可追溯性与 SHA 映射；若子目录导入重写 commit，不声称 SHA 相同。
- 标签：旧标签原地保留，新标签产品命名空间化。
- Issue：旧 URL 保留；如复制则双向链接，不声称编号不变。
- Release：旧 Release 和资产继续可访问。
- 包与更新：公开包 URL 不变；已发布客户端继续兼容 Feed。
- 源码回滚、发布权回滚、停止分发、已安装客户端恢复分别测试和记录；四者不得混称为“回滚”。

## 11. 执行交付物与验证证据

每个 Gate 产生可审阅文件，不以口头结论通过：

- 命名注册表与禁止用语清单；
- 每产品 source/release authority 状态表；
- 产品身份与外部 URL 清单；
- legacy SHA 映射与导入报告；
- 所有实际命令、环境、退出码与制品 SHA-256；
- CI 影响图及 Fixture 结果；
- Rustzen Pro 协议矩阵与兼容性测试结果；
- 反向导出、发布权恢复、停止分发和客户端恢复演练；
- PoC 前后成本与 CI 指标对比；
- 模板来源/依赖台账、泄漏扫描和删除示例后的验证结果；
- 每个 Stop/Go 决策的批准人、时间和依据。

任何附加共享消费者、产品或发布渠道都必须拥有自己的准入、验证与切换 Gate，不能沿用前一个产品的结论自动通过。

## 12. 授权边界与下一步

本 RFC 获批后，默认只授权只读准备工作，不授权任何仓库、凭据、发布者、外部注册表、分支保护、可见性或源码权威变更。

### Phase A：RFC 获批即授权的只读工作

1. 固化命名注册表草案；不改现有公开品牌和仓库名。
2. 读取并记录 G0 清单，保护并记录所有现有工作区状态，尤其是 Clear。
3. 收集 30 天维护、CI 和发布基线，起草 measurement registry 与阈值。
4. 起草 `rustzen-desktop-template` 的公开来源 allowlist、README 定位和许可候选；不复制源码、不创建仓库。

### Phase B：每项都需要单独明确授权

- 创建一次性私有 PoC 及复制源码/历史；
- 创建正式 `rustzen-tools`；
- 执行 G4，或把旧仓库设为冻结、受控镜像或只读；
- 执行 G5，或修改 npm/OIDC、签名、更新 Feed、外部注册表或凭据权限；
- 创建 `rustzen-desktop-template`、选择/变更其许可证、设为公开或发布；
- 归档或删除旧仓库。

任何 Phase B 授权只覆盖被点名的动作、产品、环境和 Gate，不自动授权后续 Gate 或其他产品。
